## 0.2 本版修订摘要

- 后端实际框架为 `FastAPI`，不再是旧版文档中的 `Flask`
- 前端状态管理实际为 `Pinia`，不再是 `Vuex`
- 前端组件库实际为 `Element Plus`，不再混写 `Ant Design`
- 前端已经不是单一页面，当前有两个路由：
  - `/`：四步工作台
  - `/fixed-rules`：固定规则检查页
- 步骤 3 已与 `/fixed-rules` 对齐为「规则组 + 规则列表 + 规则弹窗」编排区，数据仅存 `workbench` store；动态规则配置 UI 仍隐藏
- 新增固定规则模块：
  - 接口前缀：`/api/v1/fixed-rules/*`
  - 单独路由：`/fixed-rules`
  - 配置持久化：`backend/.runtime/fixed-rules/default.json`
  - 当前配置结构版本：`version = 4`
  - 固定规则页拥有自己的 `sources / variables / groups / rules` 持久化配置
  - 规则绑定改为 `target_variable_tag`，当前固定规则执行只消费单变量
  - 固定规则页当前支持 `fixed_value_compare / not_null / unique`
  - 新增规则时，`规则名称` 会按 `sheet-目标列-规则选择名称(+值)` 自动生成，并在用户未手改时随表单同步
  - 固定规则页顶部复用了工作台的数据源接入管理与变量池构建 UI，但底层状态与主工作台完全隔离
- 当前固定规则模块的 `svn` 能力已支持显式配置、`PATH` 与 Windows TortoiseSVN 自动探测，并已在有效工作副本上完成真实更新
- 当前 `feishu` 读取仍为占位实现
- 变量池复用面板中的中文文案已恢复，主工作台与固定规则页不再显示 `????` 占位文本
- 固定规则配置读取现在会把“本地路径失效 / Sheet 不存在 / 列不存在”作为非阻断问题返回到 `meta.config_issues`，固定规则页可继续加载并展示中文 warning；保存和执行仍保持明确中文 `400`

## 0.3 多用户架构需求（已实现）

- **账号与角色体系**：区分 **超级管理员（Super Admin）**、**项目管理员（Project Admin）**、**普通用户（Normal User）**，用于控制台权限与数据边界控制。
- **注册与登录**：用户通过注册 / 登录获取 **JWT**，后续 API 请求携带令牌完成鉴权。
- **默认管理员策略**：系统启动时固定维护一个默认全局超级管理员账户 `admin / 123456`；普通注册用户默认均为普通用户，不再通过注册流程自动晋升为超级管理员。
- **运行时数据库初始化**：系统启动时会自动初始化运行时数据库表结构，并确保当前运行时数据库 [D:\project\excel-check\backend\.runtime\excel_check.db](D:/project/excel-check/backend/.runtime/excel_check.db) 至少具备：
  - 默认项目 `默认项目`
  - 默认超级管理员 `admin / 123456`
  - 默认管理员在默认项目中的项目管理员角色
- **管理控制台**：
  - **超级管理员** 负责全局项目创建、修改、删除，以及用户管理、角色分配与系统配置。
  - **项目管理员** 负责项目内的数据源、规则、成员与工作台配置治理，不具备全局项目删除能力。
- **默认项目保护**：系统保留默认项目 `默认项目`，超级管理员当前也不能删除该项目。
- **项目删除成员迁移**：删除普通项目时，项目内成员不会删除账号，而是自动迁移到默认项目，并统一降为普通用户；若用户已在默认项目存在角色记录，则保留原记录不覆盖。
- **后台删除刷新策略**：管理后台删除项目成功后执行整页刷新，确保项目列表、成员区、当前项目上下文和用户项目状态全部重载。
- **成员删除规则**：
  - 默认项目中删除成员 = 删除用户账号
  - 其他项目中删除成员 = 迁移到默认项目
  - 默认项目中的超级管理员与当前登录管理员本人不可删除
- **删除项目响应兼容**：管理后台删除项目接口保持 `204 No Content` 语义，前端统一请求封装需要兼容空响应体，不得因空响应再次抛出 JSON 解析错误。
- **数据隔离**：
  - **项目级**：固定规则等共享资源按 **`project_id`** 隔离（同项目成员看到同一套项目空间配置）。
  - **页签 / 工作台级**：主工作台编排与状态按 **`project_id` + `user_id`** 隔离，每位用户在项目内拥有独立的个人工作台空间。
- **工作台自动保存**：主工作台状态支持 **2 秒防抖** 的后台自动持久化，降低手动保存成本并避免高频写入。
- **切换项目数据同步**：用户在个人设置或顶栏切换项目后，工作台与固定规则页数据自动与新项目对齐（SPA 内 store 重载），不再依赖整页刷新，空项目不残留上一项目数据。
- **空间语义**：
  - **固定规则**：归属项目的 **公共空间**（项目内共享的规则与数据源配置）。
  - **主工作台（workbench）**：归属用户的 **私有空间**（个人编排与执行上下文，与同项目其他用户互不覆盖）。

## 1. 系统现状总览

## 1.1 产品定位

本系统是一个本地运行的配置表校验工作台，面向策划表、配置表、掉落表、商店表等 Excel/CSV 数据源的规则校验场景。当前实现聚焦两个使用入口：

1. 四步工作台：适合临时编排数据源、变量和规则后立即执行
2. 固定规则检查页：适合长期维护固定规则页自己的数据源、变量池、规则组和固定规则后重复执行

## 1.2 当前真实技术栈

| 分层 | 当前实现 | 说明 |
|:---|:---|:---|
| 前端框架 | Vue 3 + TypeScript | 单页应用 |
| 前端路由 | Vue Router | 当前 2 个主路由 |
| 前端状态 | Pinia | `workbench` 与 `fixed-rules` 两套 Store |
| 前端组件库 | Element Plus | 表单、弹窗、表格、分页、消息提示 |
| 前端构建 | Vite | 本地开发代理到 `127.0.0.1:8000` |
| 后端框架 | FastAPI | REST API、Pydantic 校验 |
| 数据处理 | Pandas | Excel/CSV 读取与规则计算 |
| Excel 依赖 | `openpyxl` / `xlrd` | `.xlsx` 用 `openpyxl`，`.xls` 用 `xlrd` |
| 本地文件选择 | `tkinter` | 由后端弹出系统文件选择框 |
| SVN 调用 | `subprocess` + CLI | 当前依赖本机 `svn.exe` 或 `PATH` |

## 1.3 当前能力边界

### 已落地

- 本地 Excel / CSV 数据源接入
- Excel 元数据读取
- 变量池构建
- 组合变量 JSON 预览与持有
- 主工作台步骤 3 编排并提交引擎的规则类型：
  - `fixed_value_compare`
  - `not_null`
  - `unique`
  - `composite_condition_check`
- 引擎仍支持 `cross_table_mapping`；主工作台步骤 3 UI 不提供跨表映射表单（可通过直接调用 `POST /api/v1/engine/execute` 提交）
- 固定规则模块执行：
  - `fixed_value_compare`
  - `not_null`
  - `unique`
  - 其中比较型规则继续使用操作符 `eq / ne / gt / lt`
- 固定规则页自己的数据源接入管理与变量池构建
- 固定规则页 `version = 4` 持久化配置与旧版 `version = 3` 自动迁移
- 统一结果看板
- 固定规则配置持久化与自动回填
- 固定规则规则组、规则 CRUD、按目标变量选规
- 固定规则模块的工作副本 `SVN 更新`

### 已保留接口但未完全打通

- `regex` 规则类型已注册，但当前仍返回空结果，占位未闭环
- `svn` 作为主工作台独立 `source` 类型的完整闭环仍未开放；当前已打通的是固定规则模块的工作副本更新
- `feishu` 读取入口已保留，但加载器仍抛出 `NotImplementedError`

### 当前明确未开放

## 3. 前端详细设计

## 3.1 应用外壳

`App.vue` 当前提供统一头部导航，包含：

- 品牌区：`Excel Check`
- 导航项：
  - 工作台
  - 固定规则检查

路由定义如下：

```ts
routes: [
  { path: '/', component: MainBoard },
  { path: '/fixed-rules', component: FixedRulesBoard },
]
```

## 3.2 主工作台页面设计

`MainBoard.vue` 采用四段式工作流设计。

### 步骤 1：数据源接入

由 `DataSourcePanel.vue` 负责。

当前职责：

- 新增、编辑、删除数据源
- 调用本地文件选择接口
- 保存本地 Excel / CSV 路径
- 清理相关元数据缓存

### 步骤 2：变量池构建

由 `VariablePoolPanel.vue` 负责。

当前支持两类变量：

- 单变量 `variable_kind = single`
- 组合变量 `variable_kind = composite`

单变量字段：

- `source_id`
- `sheet`
- `column`
- `tag`
- `expected_type`

组合变量字段：

- `source_id`
- `sheet`
- `columns`
- `key_column`
- `tag`
- `expected_type = json`

说明：

- 组合变量当前只支持同一数据源、同一 Sheet、多列组合
- 组合变量会进入 `TaskTree.variables`
- 组合变量可参与步骤 3 的 `composite_condition_check` 编排；`not_null` / `unique` / `fixed_value_compare` 仅面向单变量

### 步骤 3：规则编排

由 `WorkbenchRuleOrchestrationPanel.vue` 负责（与 `FixedRulesBoard.vue` 中规则组/规则区交互同构，状态隔离）。

当前现状：

- 规则组导航、当前组规则表、分页与「新增规则 / 编辑规则」弹窗与固定规则页一致
- 编排结果映射为 `ValidationRule[]` 后经 `buildTaskTreePayload` 提交 `POST /api/v1/engine/execute`
- 工作台不持久化编排配置（刷新丢失）；与 `fixed-rules` JSON 无任何读写交叉

当前工作台步骤 3 可编排规则类型：

- `fixed_value_compare`
- `not_null`
- `unique`
- `composite_condition_check`

### 步骤 4：结果看板

由 `ResultBoardPanel.vue` 负责。

当前展示：

- 扫描总行数
- 执行耗时
- failed sources
- 异常结果表

## 3.3 固定规则页面设计

`FixedRulesBoard.vue` 当前已调整为“上下结构”。

### 顶部操作条

固定展示以下按钮：

- 保存配置
- SVN 更新
- 执行全部规则

并展示整体统计：

- 规则组数量
- 规则总数
- 固定路径数
- 异常结果数

### 规则组导航带

上方横向展示规则组胶囊导航，包含：

- 搜索框
- 新建规则组
- 每组规则数量 badge
- 失效组提示
- 当前激活态

规则组规则：

- 默认组：`未分组`
- 默认组不可删除、不可改名
- 删除自定义组时，组内规则自动迁移到 `未分组`

### 当前组规则区

下方展示当前选中组的规则列表。

每条规则直接展示：

- 规则名称
- 绑定文件摘要
- `sheet / column`
- 规则选择摘要
- 条件值（仅比较类规则）

支持：

- 新增规则
- 编辑规则
- 删除规则
- 分页展示

分页参数：

- 固定每页 `20` 条

### 规则编辑弹窗

每次新增或编辑规则时，必须单独配置：

- 规则组
- 规则名称
- 目标变量
- 规则选择
- 比较值（仅比较类规则显示）

当前不再手工维护规则级文件路径，而是直接从固定规则页变量池选择目标变量。
选中目标变量后，页面会展示只读摘要：

- 来源数据源
- Sheet
- 列名
- 路径

规则名称交互规则：

- 新增规则时，默认按 `sheet-目标列-规则选择名称(+值)` 自动生成
- 比较类规则带 `+值`，非比较类不带
- 只要用户还未手动修改规则名称，`目标变量 / 规则选择 / 比较值` 变化后会自动刷新默认名
- 一旦用户手动改名或主动清空规则名称，后续字段变化不再自动覆盖
- 规则名称仍为必填，空规则名不能保存

当前规则选择项固定为：

- `等于 (=)`
- `不等于 (!=)`
- `大于 (>)`
- `小于 (<)`
- `非空校验`
- `唯一校验`

其中：

- 比较类规则保存为 `rule_type = fixed_value_compare`
- `非空校验` 保存为 `rule_type = not_null`
- `唯一校验` 保存为 `rule_type = unique`
- `not_null / unique` 不展示也不保存 `expected_value`
- 当前固定规则执行只允许绑定单变量；组合变量继续保留在固定规则页变量池中做持久化、编辑和预览

## 4.4 执行引擎设计

`execute_api.py` 是主工作台的统一执行入口。

主流程：

1. 校验 `TaskTree`
2. 按 `source_id` 聚合变量
3. 用线程池并发加载数据源
4. 对失败的数据源做降级记录
5. 过滤依赖失败源的规则
6. 调用规则注册表执行规则
7. 统一组装返回结构

### 并发加载策略

当前使用：

- `ThreadPoolExecutor(max_workers=settings.default_thread_pool_size)`

适用场景：

- 多数据源并发读取
- SVN 同步 + 本地文件读取

### 规则执行入口

`engine_core.py` 当前使用注册表模式：

```python
RULE_REGISTRY: dict[str, RuleHandler] = {}
```

执行时按 `task_tree.rules` 顺序遍历：

1. 查找 `rule_type`
2. 获取 handler
3. 传入 `RuleExecutionContext`
4. 汇总 `abnormal_results`

## 4.5 固定规则服务层设计

`fixed_rules/service.py` 是固定规则模块的核心服务层。

主要职责：

- 构建默认配置
- 读取持久化配置
- 兼容旧版配置迁移
- 校验与标准化配置
- 生成临时 `TaskTree`
- 触发执行
- 聚合固定路径的 SVN 更新

### 当前配置版本

- 当前版本：`version = 4`

### 当前持久化路径

- `backend/.runtime/fixed-rules/default.json`

### 旧版迁移策略

- 若发现旧版顶层字段：
  - `file_path`
  - `sheet`
  - `columns`
  - `svn_enabled`
  则先迁移为规则级结构，再继续规范化为当前 `version = 4` 配置。
- 若检测到 `version = 3` 或规则仍使用 `binding.file_path / binding.sheet / binding.column`
  - 服务层会自动去重生成 `sources`
  - 自动生成或复用对应单变量 `variables`
  - 将规则改写为 `target_variable_tag`
  - 最终统一保存为 `version = 4`

### 固定规则执行编排

固定规则不会复刻第二套执行引擎，而是：

1. 从已保存配置读取规则
2. 根据规则引用的 `target_variable_tag` 聚合出临时 `sources`
3. 根据规则引用的 `target_variable_tag` 聚合出临时 `variables`
4. 按组顺序和规则顺序生成 `ValidationRule`
5. 复用现有 `execute_engine`

这样可以确保：

- 主工作台和固定规则页使用相同结果结构
- 规则执行逻辑只维护一套

```json
{
```

```json
  "sources": [
    {
      "id": "fixed-src-demo",
      "type": "local_excel",
      "path": "D:/project/excel-check/.codex/runtime/fixed_rules_flow.xlsx"

      "column": "INT_ID",
      "expected_type": "str"
    },
    {
      "tag": "[fixed-src-demo-items-INT_ID-mapping]",
      "source_id": "fixed-src-demo",

      "expected_type": "json"
    }
  ],

    {
      "group_id": "ungrouped",
      "group_name": "未分组",
      "builtin": true
    },
    {
      "group_id": "basic-checks",
      "group_name": "基础校验",
      "builtin": false
    }
  ],
  "rules": [
    {
      "rule_id": "rule-int-id-gt-0",
      "group_id": "basic-checks",
      "rule_name": "INT_ID 必须大于 0",
      "target_variable_tag": "[fixed-src-demo-items-INT_ID]",
      "rule_type": "fixed_value_compare",
      "operator": "gt",
      "expected_value": "0"
    },
    {
      "rule_id": "rule-int-id-not-null",
      "group_id": "basic-checks",

      "target_variable_tag": "[fixed-src-demo-items-INT_ID]",

```

说明：

- `sources / variables / groups / rules` 统一由 `/api/v1/fixed-rules/config` 持久化

- `not_null / unique` 不需要 `operator` 与 `expected_value`

执行时按以下顺序展开：

1. 先按 `groups` 数组顺序
2. 再按该组下规则在 `rules` 数组中的出现顺序

### 固定规则内部变量引用

后端执行时会直接复用已保存的 `VariableTag.tag`，不再额外生成新的内部文件绑定标签。

## 7. 规则详细设计

## 7.1 `not_null`

作用：

- 检查单变量列中的空值

输入：

- `params.target_tags: string[]`

输出：

- `level = error`

## 7.2 `unique`

作用：

- 检查单变量列中的重复值

输入：

- `params.target_tags: string[]`

输出：

- `level = warning`

## 7.3 `cross_table_mapping`

作用：

- 检查目标列中的值是否都出现在基础字典列中

输入：

- `params.dict_tag`
- `params.target_tag`

输出：

- `level = error`

## 7.4 `fixed_value_compare`

作用：

- 固定规则模块的单列常量比较规则

输入：

- `params.target_tag`
- `params.operator`
- `params.expected_value`
- `params.rule_name`

操作符语义：

| 操作符 | 语义 |
|:---|:---|
| `eq` | 去首尾空白后精确等于 |
| `ne` | 去首尾空白后精确不等于 |
| `gt` | 数值大于 |
| `lt` | 数值小于 |

补充说明：

- `eq / ne` 用文本比较
- `gt / lt` 用数值比较
- 若数值比较遇到“非空但不可转数值”的单元格，会返回异常而不是跳过
- 固定规则页当前也可直接编排 `not_null` 与 `unique`，这两类规则复用基础规则实现，并分别保持 `error / warning` 语义

## 7.5 `regex`

当前状态：

- 已注册
- 暂未实现真实计算
- 当前返回空列表

## 8. 关键执行时序

## 8.1 主工作台时序

1. 用户在步骤 1 保存数据源
2. 用户在步骤 2 构建单变量或组合变量
3. 用户在步骤 3 通过规则组与规则弹窗编排规则并保存到工作台内存态
4. 前端调用 `buildTaskTreePayload`
5. 前端请求 `POST /api/v1/engine/execute`
6. 后端按 `source_id` 聚合变量
7. 后端线程池并发加载数据源
8. 后端过滤依赖失败源的规则
9. 规则引擎按顺序执行规则
10. 后端组装统一结果响应
11. 前端渲染步骤 4 结果看板

## 8.2 固定规则页时序

1. 页面打开调用 `GET /api/v1/fixed-rules/config`
2. Store 自动回填固定规则页自己的数据源、变量池、规则组和规则
3. 用户先维护固定规则页数据源与变量，再在弹窗中为单条规则选择目标变量、规则类型和比较值
4. 前端调用 `PUT /api/v1/fixed-rules/config` 保存配置
5. 用户点击“执行全部规则”
6. 前端先保存配置，再调用 `POST /api/v1/fixed-rules/execute`
7. 后端根据 `target_variable_tag` 聚合临时 `TaskTree`
8. 后端复用主执行引擎
9. 返回统一结果响应
10. 前端在固定规则页结果区渲染摘要和异常列表

## 8.3 固定规则配置迁移时序

1. 后端读取 `default.json`
2. 若检测到旧版顶层 `file_path / sheet / columns / svn_enabled`
3. 自动映射为过渡规则级结构
4. 再继续迁移为 `sources / variables / rules[*].target_variable_tag`
5. 最终按 `version = 4` 执行统一校验与标准化
5. 返回新版配置给前端

| `.xlsx` 依赖缺失 | 提示安装 `openpyxl` |
| 规则引用未知 tag | 返回 `400` |
| 规则依赖失败数据源 | 规则被过滤，不阻断整次执行 |
| 固定规则列不存在 | 保存配置时直接失败 |
| `svn` CLI 缺失 | 返回明确安装/配置提示 |
| 目标目录不是 SVN 工作副本 | 返回明确错误 |
| 飞书读取 | 当前直接 `NotImplementedError` |

## 10.2 当前性能策略

- Excel / CSV 读取优先使用 `usecols`
- 多数据源读取采用线程池并发
- 规则执行采用 Pandas 列级运算
- 统一结果结构避免重复转换成本

## 10.3 当前安全与运行约束

- 本地文件选择依赖桌面环境和 `tkinter`
- 当前更适合本机开发机或桌面联调环境
- 若部署到无桌面的服务环境，`local-pick` 不适用
