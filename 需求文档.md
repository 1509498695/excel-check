## 1. 系统现状总览

## 1.1 产品定位

本系统是一个本地运行的配置表校验工作台，面向策划表、配置表、掉落表、商店表等 Excel/CSV 数据源的规则校验场景。当前实现聚焦两个使用入口：

1. 四步工作台：适合临时编排数据源、变量和规则后立即执行
2. 固定规则检查页：适合长期维护固定文件、固定规则组和固定规则后重复执行

## 1.2 当前真实技术栈

| 分层 | 当前实现 | 说明 |
|:---|:---|:---|
| 前端框架 | Vue 3 + TypeScript | 单页应用 |
| 前端路由 | Vue Router | 当前 2 个主路由 |
| 前端状态 | Pinia | `workbench` 与 `fixed-rules` 两套 Store |
| 前端组件库 | Element Plus | 表单、弹窗、表格、分页、消息提示 |
| 前端构建 | Vite | 本地开发代理到 `127.0.0.1:8000` |
| 后端框架 | FastAPI | REST API、Pydantic 校验 |
| 数据处理 | Pandas | Excel/CSV 读取与规则计算 |
| Excel 依赖 | `openpyxl` / `xlrd` | `.xlsx` 用 `openpyxl`，`.xls` 用 `xlrd` |
| 本地文件选择 | `tkinter` | 由后端弹出系统文件选择框 |
| SVN 调用 | `subprocess` + CLI | 当前依赖本机 `svn.exe` 或 `PATH` |

## 1.3 当前能力边界

### 已落地

- 本地 Excel / CSV 数据源接入
- Excel 元数据读取
- 变量池构建
- 组合变量 JSON 预览与持有
- 静态规则执行：
  - `not_null`
  - `unique`
  - `cross_table_mapping`
- 固定规则模块执行：
  - `fixed_value_compare`
  - 操作符 `eq / ne / gt / lt`
- 统一结果看板
- 固定规则配置持久化与自动回填
- 固定规则规则组、规则 CRUD、按规则绑定文件

### 已保留接口但未完全打通

- `regex` 规则类型已注册，但当前仍返回空结果，占位未闭环
- `svn` 相关能力已具备后端调用和错误提示，但当前环境未检测到 CLI 时只能降级
- `feishu` 读取入口已保留，但加载器仍抛出 `NotImplementedError`

### 当前明确未开放

- 步骤 3 的动态规则配置 UI
- 固定规则多配置集切换
- 固定规则结果导出
- CSV 的变量池元数据下拉提取

## 3. 前端详细设计

## 3.1 应用外壳

`App.vue` 当前提供统一头部导航，包含：

- 品牌区：`Excel Check`
- 导航项：
  - 工作台
  - 固定规则检查

路由定义如下：

```ts
routes: [
  { path: '/', component: MainBoard },
  { path: '/fixed-rules', component: FixedRulesBoard },
]
```

## 3.2 主工作台页面设计

`MainBoard.vue` 采用四段式工作流设计。

### 步骤 1：数据源接入

由 `DataSourcePanel.vue` 负责。

当前职责：

- 新增、编辑、删除数据源
- 调用本地文件选择接口
- 保存本地 Excel / CSV 路径
- 清理相关元数据缓存

### 步骤 2：变量池构建

由 `VariablePoolPanel.vue` 负责。

当前支持两类变量：

- 单变量 `variable_kind = single`
- 组合变量 `variable_kind = composite`

单变量字段：

- `source_id`
- `sheet`
- `column`
- `tag`
- `expected_type`

组合变量字段：

- `source_id`
- `sheet`
- `columns`
- `key_column`
- `tag`
- `expected_type = json`

说明：

- 组合变量当前只支持同一数据源、同一 Sheet、多列组合
- 组合变量会进入 `TaskTree.variables`
- 但当前静态规则不消费组合变量

### 步骤 3：规则编排

由 `RuleComposerPanel.vue` 负责。

当前现状：

- 页面只展示静态规则模板
- 动态规则配置 UI 已隐藏
- 执行请求只会提交静态规则
- `mode === 'dynamic'` 的数据结构兼容仍保留在前端 Store 中，但不进入最终执行

当前可见静态规则模板：

- `not_null`
- `unique`
- `cross_table_mapping`

### 步骤 4：结果看板

由 `ResultBoardPanel.vue` 负责。

当前展示：

- 扫描总行数
- 执行耗时
- failed sources
- 异常结果表

## 3.3 固定规则页面设计

`FixedRulesBoard.vue` 当前已调整为“上下结构”。

### 顶部操作条

固定展示以下按钮：

- 保存配置
- SVN 更新
- 执行全部规则

并展示整体统计：

- 规则组数量
- 规则总数
- 固定路径数
- 异常结果数

### 规则组导航带

上方横向展示规则组胶囊导航，包含：

- 搜索框
- 新建规则组
- 每组规则数量 badge
- 失效组提示
- 当前激活态

规则组规则：

- 默认组：`未分组`
- 默认组不可删除、不可改名
- 删除自定义组时，组内规则自动迁移到 `未分组`

### 当前组规则区

下方展示当前选中组的规则列表。

每条规则直接展示：

- 规则名称
- 绑定文件摘要
- `sheet / column`
- 操作符
- 目标值

支持：

- 新增规则
- 编辑规则
- 删除规则
- 分页展示

分页参数：

- 固定每页 `20` 条

### 规则编辑弹窗

每次新增或编辑规则时，必须单独配置：

- 规则组
- 规则名称
- 文件路径
- Sheet
- 目标列
- 操作符
- 比较值

文件配置是规则级绑定，不再有全局固定文件配置卡。

当前行为：

1. 创建 `FastAPI` 实例
2. 注册 `api_router`
3. 暴露 `/health`

`backend/config.py` 当前核心配置：

| 字段 | 当前值 / 作用 |
|:---|:---|
| `app_name` | `excel-check-backend` |
| `host` | `127.0.0.1` |
| `port` | `8000` |
| `api_v1_prefix` | `/api/v1` |
| `default_thread_pool_size` | `4` |
| `fixed_rules_config_path` | `backend/.runtime/fixed-rules/default.json` |
| `svn_executable` | 默认 `svn` |
| `supported_source_types` | `local_excel / local_csv / feishu / svn` |

## 4.2 API 路由装配

`backend/app/api/router.py` 当前聚合三个子路由：

- `source_router`
- `execute_router`
- `fixed_rules_router`

## 4.3 数据读取层设计

### `local_reader.py`

职责：

- 读取本地 Excel / CSV
- 提取 Excel 元数据
- 预览单列数据
- 预览组合变量 JSON
- 按变量切片装载 DataFrame

关键实现：

- `.xlsx` 使用 `openpyxl`
- `.xls` 使用 `xlrd`
- 通过 `usecols` 只读取目标列，控制内存
- 单变量载入时统一追加 `_row_index = index + 2`
- 组合变量会构造成 `json_value` 单列表结构

### `svn_manager.py`

职责：

- 探测 `svn` 可执行文件
- 调用 `svn update`
- 对“不是工作副本”“CLI 缺失”等情况给出明确错误

当前特点：

- 依赖 `settings.svn_executable`
- 若 `PATH` 中找不到 `svn`，会抛出明确降级提示
- 固定规则页的 SVN 更新最终也走这里

### `feishu_reader.py`

当前状态：

- 占位实现
- 当前直接抛出 `NotImplementedError`

## 4.4 执行引擎设计

`execute_api.py` 是主工作台的统一执行入口。

主流程：

1. 校验 `TaskTree`
2. 按 `source_id` 聚合变量
3. 用线程池并发加载数据源
4. 对失败的数据源做降级记录
5. 过滤依赖失败源的规则
6. 调用规则注册表执行规则
7. 统一组装返回结构

### 并发加载策略

当前使用：

- `ThreadPoolExecutor(max_workers=settings.default_thread_pool_size)`

适用场景：

- 多数据源并发读取
- SVN 同步 + 本地文件读取

### 规则执行入口

`engine_core.py` 当前使用注册表模式：

```python
RULE_REGISTRY: dict[str, RuleHandler] = {}
```

执行时按 `task_tree.rules` 顺序遍历：

1. 查找 `rule_type`
2. 获取 handler
3. 传入 `RuleExecutionContext`
4. 汇总 `abnormal_results`

## 4.5 固定规则服务层设计

`fixed_rules/service.py` 是固定规则模块的核心服务层。

主要职责：

- 构建默认配置
- 读取持久化配置
- 兼容旧版配置迁移
- 校验与标准化配置
- 生成临时 `TaskTree`
- 触发执行
- 聚合固定路径的 SVN 更新

### 当前配置版本

- 当前版本：`version = 2`

### 当前持久化路径

- `backend/.runtime/fixed-rules/default.json`

### 旧版迁移策略

若发现旧版顶层字段：

- `file_path`
- `sheet`
- `columns`
- `svn_enabled`

则自动迁移为新版“规则级 binding”结构。

### 固定规则执行编排

固定规则不会复刻第二套执行引擎，而是：

1. 从已保存配置读取规则
2. 聚合出临时 `sources`
3. 聚合出临时 `variables`
4. 按组顺序和规则顺序生成 `ValidationRule`
5. 复用现有 `execute_engine`

这样可以确保：

- 主工作台和固定规则页使用相同结果结构
- 规则执行逻辑只维护一套

```json
{
  "id": "src_demo",

  "pathOrUrl": "D:/path/demo.xlsx",
```

字段说明：

| 字段 | 说明 |
|:---|:---|

| `pathOrUrl` | 兼容字段 |
| `url` | 飞书地址 |
| `token` | 飞书 token |

### VariableTag

```json
{
  "tag": "[src_items-items-INT_ID]",
  "source_id": "src_items",
  "sheet": "items",
  "variable_kind": "single",
  "column": "INT_ID",
  "columns": null,
  "key_column": null,
  "expected_type": "str"
}
```

### ValidationRule

```json
{
  "rule_id": "rule-1",
  "rule_type": "cross_table_mapping",
  "params": {
    "dict_tag": "[items-id]",
    "target_tag": "[drops-ref]"
  }
}
```

### TaskTree

```json
{
  "sources": [],
  "variables": [],
  "rules": []
}
```

## 5.2 固定规则模块协议

当前固定规则协议为规则级文件绑定。

```json
{
  "version": 2,
  "configured": true,
  "groups": [
    {
      "group_id": "ungrouped",
      "group_name": "未分组",
      "builtin": true
    },
    {
      "group_id": "basic-checks",
      "group_name": "基础校验",
      "builtin": false
    }
  ],
  "rules": [
    {
      "rule_id": "rule-int-id-gt-0",
      "group_id": "basic-checks",
      "rule_name": "INT_ID 必须大于 0",
      "binding": {
        "file_path": "D:/projact_samo/GameDatas/datas_qa88/items.xls",
        "sheet": "items",
        "column": "INT_ID"
      },
      "operator": "gt",
      "expected_value": "0"
    }
  ]
}
```

### 固定规则排序规则

执行时按以下顺序展开：

1. 先按 `groups` 数组顺序
2. 再按该组下规则在 `rules` 数组中的出现顺序

### 固定规则内部变量标签

后端执行时自动生成内部变量标签：

```text
[fixed-{source_id}-{sheet}-{column}]
```

## 7. 规则详细设计

## 7.1 `not_null`

作用：

- 检查单变量列中的空值

输入：

- `params.target_tags: string[]`

输出：

- `level = error`

## 7.2 `unique`

作用：

- 检查单变量列中的重复值

输入：

- `params.target_tags: string[]`

输出：

- `level = warning`

## 7.3 `cross_table_mapping`

作用：

- 检查目标列中的值是否都出现在基础字典列中

输入：

- `params.dict_tag`
- `params.target_tag`

输出：

- `level = error`

## 7.4 `fixed_value_compare`

作用：

- 固定规则模块的单列常量比较规则

输入：

- `params.target_tag`
- `params.operator`
- `params.expected_value`
- `params.rule_name`

操作符语义：

| 操作符 | 语义 |
|:---|:---|
| `eq` | 去首尾空白后精确等于 |
| `ne` | 去首尾空白后精确不等于 |
| `gt` | 数值大于 |
| `lt` | 数值小于 |

补充说明：

- `eq / ne` 用文本比较
- `gt / lt` 用数值比较
- 若数值比较遇到“非空但不可转数值”的单元格，会返回异常而不是跳过

## 7.5 `regex`

当前状态：

- 已注册
- 暂未实现真实计算
- 当前返回空列表

## 8. 关键执行时序

## 8.1 主工作台时序

1. 用户在步骤 1 保存数据源
2. 用户在步骤 2 构建单变量或组合变量
3. 用户在步骤 3 选择静态规则模板并补齐参数
4. 前端调用 `buildTaskTreePayload`
5. 前端请求 `POST /api/v1/engine/execute`
6. 后端按 `source_id` 聚合变量
7. 后端线程池并发加载数据源
8. 后端过滤依赖失败源的规则
9. 规则引擎按顺序执行规则
10. 后端组装统一结果响应
11. 前端渲染步骤 4 结果看板

## 8.2 固定规则页时序

1. 页面打开调用 `GET /api/v1/fixed-rules/config`
2. Store 自动回填规则组和规则
3. 用户在弹窗中为单条规则配置文件路径、Sheet、列和比较值
4. 前端调用 `PUT /api/v1/fixed-rules/config` 保存配置
5. 用户点击“执行全部规则”
6. 前端先保存配置，再调用 `POST /api/v1/fixed-rules/execute`
7. 后端从规则级 binding 聚合临时 `TaskTree`
8. 后端复用主执行引擎
9. 返回统一结果响应
10. 前端在固定规则页结果区渲染摘要和异常列表

## 8.3 固定规则配置迁移时序

1. 后端读取 `default.json`
2. 若检测到旧版顶层 `file_path / sheet / columns / svn_enabled`
3. 自动映射为新版 `rules[*].binding`
4. 再执行统一校验与标准化
5. 返回新版配置给前端

## 9. 结果模型与行号规则

所有执行入口统一返回：

```json
{
  "code": 200,
  "msg": "Execution Completed",
  "meta": {
    "execution_time_ms": 0,
    "total_rows_scanned": 0,
    "failed_sources": []
  },
  "data": {
    "abnormal_results": []
  }
}
```

### `abnormal_results` 结构

| 字段 | 说明 |
|:---|:---|
| `level` | `error / warning / ...` |
| `rule_name` | 规则名，固定规则优先使用用户配置的规则名称 |
| `location` | 定位信息 |
| `row_index` | Excel 实际行号口径 |
| `raw_value` | 原始单元格值 |
| `message` | 详细说明 |

### 行号口径

当前统一按：

```text
Pandas 索引 + 2
```

原因：

- Pandas 从 0 开始
- Excel 第 1 行通常为表头

## 10. 非功能设计与异常处理

## 10.1 当前异常处理策略

| 场景 | 当前处理 |
|:---|:---|
| 本地文件不存在 | 返回 `400`，带明确中文错误 |
| `.xls` 依赖缺失 | 提示安装 `xlrd` |
| `.xlsx` 依赖缺失 | 提示安装 `openpyxl` |
| 规则引用未知 tag | 返回 `400` |
| 规则依赖失败数据源 | 规则被过滤，不阻断整次执行 |
| 固定规则列不存在 | 保存配置时直接失败 |
| `svn` CLI 缺失 | 返回明确安装/配置提示 |
| 目标目录不是 SVN 工作副本 | 返回明确错误 |
| 飞书读取 | 当前直接 `NotImplementedError` |

## 10.2 当前性能策略

- Excel / CSV 读取优先使用 `usecols`
- 多数据源读取采用线程池并发
- 规则执行采用 Pandas 列级运算
- 统一结果结构避免重复转换成本

## 10.3 当前安全与运行约束

- 本地文件选择依赖桌面环境和 `tkinter`
- 当前更适合本机开发机或桌面联调环境
- 若部署到无桌面的服务环境，`local-pick` 不适用

## 13. 结论

当前系统已经从“单一四步工作台”演进为“两条校验主链路并存”的状态：

1. `/` 工作台负责临时编排
2. `/fixed-rules` 负责长期维护固定规则

后端继续坚持统一执行模型：

- 统一 `TaskTree`
- 统一规则注册表
- 统一响应格式

固定规则模块当前采用“规则级文件绑定 + 单配置持久化 + 一键执行全部规则”的设计，已经是项目中最接近业务长期使用形态的模块。后续演进建议优先围绕 `SVN`、`regex`、导出和多配置集展开，而不是重新拆分执行引擎。
