# Codex 开发执行指南

文档更新时间：2026-04-02 12:26

## 目的

本文档用于固化后续使用 Codex 推进 `excel-check` 项目开发的执行方式。当前项目已经处于“前后端最小闭环可联调”阶段，后续不需要重做架构，而是要基于现有 `README.md`、`PROJECT_RECORD.md`、`CHANGELOG.md`、`frontend/README.md` 和 SDD 按纵向切片持续补齐企业级能力。

## 每次开始前必须做的事

1. 先阅读以下文档：
   - `README.md`
   - `PROJECT_RECORD.md`
   - `CHANGELOG.md`
   - `frontend/README.md`
   - 桌面 SDD 文档《企业级配置表自动化检查与动态规则校验工作台 - 系统详细设计说明书 (SDD).md》
2. 忽略以下目录或文件：
   - `frontend/node_modules`
   - `frontend/dist`
   - `__pycache__`
   - `backend-server.log`
   - `frontend-server.log`
3. 每次只选一个明确切片，不要把两个里程碑混在一次任务里完成。

## 按网页模块管理时的默认拆法

如果你希望按网页显示的模块管理排期，默认采用下面这 4 个页面模块：

1. 数据源读取和变量池
2. 正则校验
3. 动态规则检查和看板
4. 静态规则检查和看板

这里要明确区分“页面模块划分”和“实际开发顺序”：

- 页面模块可以按上面 4 个展示。
- 实际开发顺序推荐采用 `模块 1 -> 模块 4 -> 模块 2 -> 模块 3`。
- `工程基线` 不属于这 4 个页面模块，它是跨模块准备项，可在模块 1 前单独插入。

这些模块默认遵循同一套技术边界：

- 保持一个共享工作台，不拆成 4 个独立应用。
- 保持一个共享状态中心，继续复用 `TaskTree`。
- 保持一个统一执行入口 `POST /api/v1/engine/execute`。
- 保持一个统一结果看板，不复制多套结果展示系统。

### 模块 1：数据源读取和变量池

- 目标：稳定产出 `sources` 和 `variables`，为后续所有规则模块提供底座。
- 范围：本地 Excel/CSV source、变量标签、source 与 variable 绑定关系、删除依赖后的规则清理。
- 边界：这一层不承担结果看板逻辑，也不强行实现 Feishu/SVN 的完整能力。

### 模块 4：静态规则检查和看板

- 目标：先做出最短业务闭环。
- 范围：`not_null`、`unique`、`cross_table_mapping` 的编排、执行和统一结果展示。
- 边界：继续复用统一结果看板，不单独复制一套静态规则结果页。

### 模块 2：正则校验

- 目标：把 `regex` 做成第一条新增规则扩展链。
- 范围：前端规则入口、参数校验、后端规则注册与执行、统一结果展示。
- 边界：页面可单独展示，但底层仍走统一规则引擎和统一执行入口。

### 模块 3：动态规则检查和看板

- 目标：逐步把规则能力平台化。
- 范围：`rule_type + params` 编辑、JSON 校验、未注册规则报错、已注册规则执行、统一看板复用。
- 边界：第一阶段不追求“任意规则全支持”，先把最基本的输入校验和执行闭环做稳。

## 任务规模判定

### 可直接让 Codex 实现的小切片

- `regex` 规则端到端闭环
- 前端结果导出第一版
- 结果筛选或搜索
- 单个接口的错误处理补齐
- 单个占位规则或占位入口的最小实现

### 先让 Codex 出计划再进入实现的大切片

- Feishu 最小接入
- SVN 最小同步与冲突降级
- 登录、权限和用户态
- 拖拽式编排
- 工程化配置体系与 CI
- 审计日志与发布流程

## 推荐里程碑顺序

### 里程碑 0：工程基线

- 目标：先补稳定护栏，降低后续改动回归成本。
- 交付：`pytest.ini`、`pyproject.toml` 或 `ruff` 配置、前端 lint / format 配置、统一脚本命令。
- 边界：只补工程能力，不改业务逻辑。
- 验收：后端测试可稳定运行，前端构建可稳定通过，文档写清新增规范。

### 里程碑 1：`regex` 规则端到端闭环

- 目标：验证“前端规则模板 -> TaskTree -> 后端规则引擎 -> 结果看板 -> 测试”整条扩展链是健康的。
- 交付：后端 `regex` 规则实现、前端规则模板与参数校验、接口测试、结果展示说明。
- 验收：至少覆盖一条命中异常和一条不命中的测试用例。

### 里程碑 2：结果导出第一版

- 目标：把结果看板变成可交付成果，而不只是页面展示。
- 交付：前端基于 `abnormal_results` 的导出能力，优先支持 CSV 或 Excel 任一种简单格式。
- 边界：第一版默认不新增后端导出接口。
- 验收：导出内容与结果表格字段一致，操作路径清晰可复现。

### 里程碑 3：Feishu 最小可测试接入

- 目标：补齐最核心的在线数据源能力。
- 交付：后端 Feishu loader 最小读取链路、失败降级、前端 source 表单联动、接口验证和文档同步。
- 验收：成功时能读数据，失败时不会拖垮整次执行，并能进入 `failed_sources`。

### 里程碑 4：SVN 最小同步与冲突降级

- 目标：补齐“本地目录 + SVN 更新”这一条企业链路。
- 交付：`svn up` 调用、冲突或锁定识别、失败 source 降级、前端提示和测试。
- 验收：同步成功可继续读取，冲突时能明确返回可解释结果。

### 里程碑 5：前端工作台可用性增强

- 目标：把当前“能联调”升级为“更适合业务使用”。
- 交付：先做结果筛选和搜索，再单独推进拖拽式编排。
- 边界：不要把筛选搜索和拖拽式编排放进同一轮任务。
- 验收：用户能更快定位异常结果，交互路径更顺畅。

### 里程碑 6：企业级基础设施

- 目标：在核心业务链路稳定后，再补权限、审计、CI 和发布流程。
- 交付：登录与权限、用户态、操作审计、CI、发版检查清单。
- 要求：这一阶段建议每个子能力都先让 Codex 输出计划，再进入实现。

## 契约与边界

- 保持 `POST /api/v1/engine/execute` 的主结构稳定，继续使用 `TaskTree -> sources / variables / rules`。
- 保持标准响应结构稳定，继续使用 `code / msg / meta / data.abnormal_results`。
- 新规则优先复用现有 `ValidationRule.rule_type`，不要新增平行接口。
- 数据源模型建议统一为：
  - 本地和 SVN 使用 `path`
  - Feishu 使用 `url` + `token`
  - `pathOrUrl` 只保留为前端兼容输入字段，不作为长期主契约
- 导出报告第一版默认不新增后端 API。
- `GET /api/v1/sources/capabilities` 后续如果增强，优先让返回内容更准确，不要贸然改成复杂新协议。
- 所有面向人的文档说明都使用中文，代码标识符、命令、路径和接口字段保持原语言。

## 每次执行的收尾清单

1. 后端有变更时运行：`python -m pytest backend/tests -q`
2. 前端有变更时运行：`cd frontend && npm run build`
3. 涉及联调时，用 `backend/tests/data/minimal_rules.xlsx` 跑一轮最小闭环
4. 更新 `README.md`，同步项目当前全貌
5. 向 `PROJECT_RECORD.md` 追加分钟级记录
6. 形成阶段里程碑时，同步更新 `CHANGELOG.md`
7. 最终总结必须写清：本次完成、验证结果、遗留风险、建议下一步

## 当前推荐下一步

- 如果从跨模块准备项开始，推荐先执行 `里程碑 0：工程基线`。
- 如果按网页模块排期，默认顺序为 `模块 1 -> 模块 4 -> 模块 2 -> 模块 3`。
- 模块 2 和模块 3 都复用统一执行入口与统一结果看板，不要分裂成独立系统。
- 4 个页面模块稳定后，再继续推进 Feishu、SVN、导出、权限、审计和持续集成流程。
